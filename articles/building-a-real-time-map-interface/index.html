Building a Real-Time Map Interface with WebGL

March 15, 2026

There's something deeply satisfying about watching objects move across a map in real time. Not the janky, marker-hopping kind of movement you get from polling an API every few seconds, but smooth, interpolated motion that feels like you're watching a live feed.

I've been building exactly this — a Command & Conquer-inspired interface that renders hundreds of units on a WebGL map layer. The project started as a weekend experiment and turned into a deep dive through rendering pipelines, coordinate systems, and the surprisingly tricky math of making things look right on a sphere.

Here's what I've learned along the way.

The Rendering Stack

Early on I tried to do everything in a single renderer. That was a mistake. Each rendering technology has a sweet spot, and the real insight was layering them so each handles what it does best:

  • Mapbox GL — base map tiles, camera control, and projection
  • deck.gl — geospatial WebGL overlays (unit models, rally lines, labels)
  • PixiJS — 2D sprite rendering and effects (selection circles, fog of war)
  • Vue.js — UI chrome (sidebar, settings panel, HUD stats)

The stack renders bottom-to-top: Mapbox draws the map, deck.gl composites its WebGL layers on top via a MapboxOverlay, PixiJS sits on a transparent canvas above that, and Vue's HTML elements float over everything.

The trick isn't picking the best renderer — it's knowing which layer each feature belongs on. A selection circle needs to track a geo-coordinate and survive map tilting, so it goes on deck.gl. A fog of war effect is pure 2D masking, so it goes on PixiJS.

Client-Side Interpolation

The server runs a simulation loop at 20Hz — every 50ms it ticks the world forward, updating positions, headings, and altitudes. But we need 60fps on the client for smooth visuals. Displaying raw server updates would look choppy.

The solution is client-side interpolation. Each frame, we lerp between the last known position and the predicted next position:

function lerpPos(
  a: Position,
  b: Position,
  t: number
): Position {
  return {
    lat: a.lat + (b.lat - a.lat) * t,
    lon: a.lon + (b.lon - a.lon) * t,
    alt: a.alt + (b.alt - a.alt) * t,
  };
}

The t parameter represents progress between two server snapshots, clamped to [0, 1] to prevent overshooting when a network update is delayed.

Heading Interpolation

Position lerp is straightforward — latitude and longitude are continuous values. But heading requires special care. If a vehicle's heading changes from 350° to 10°, naive interpolation would sweep 340° the wrong way around.

You need to find the shortest arc:

function shortestAngleDiff(from: number, to: number): number {
  const diff = ((to - from + 180) % 360) - 180;
  return diff < -180 ? diff + 360 : diff;
}

// Usage in the interpolation loop
const headingDelta = shortestAngleDiff(prev.heading, next.heading);
const heading = prev.heading + headingDelta * t;

This tiny function saved me hours of debugging vehicles that would randomly spin full circles during course corrections.

The Fog of War

This is where PixiJS earns its place in the stack. The fog of war system uses an inverse-mask approach: a full-screen dark overlay with holes punched where vehicles have been.

The implementation has two components:

  1. Live stamps — radial gradient circles rendered at each vehicle's current screen position, every frame
  2. Trail texture — a persistent RenderTexture that accumulates all previous stamps, creating a permanent record of explored terrain

Each frame, we render the trail texture plus live stamps into a mask, then apply that mask inversely to a dark overlay. Areas that have been "seen" become transparent, revealing the map beneath.

// Stamp a radial gradient at each vehicle's position
for (const unit of units) {
  const screen = map.project([unit.lon, unit.lat]);
  stamp.position.set(screen.x, screen.y);
  stamp.scale.set(revealRadius / baseSize);

  // Accumulate into the persistent trail
  renderer.render(stamp, { renderTexture: trailTexture, clear: false });
}

The visual effect is striking — as your units sweep across the map, they leave behind a permanent trace of everywhere they've been, while unexplored territory stays shrouded.

Input Handling

Getting input right in a multi-layer rendering stack is surprisingly fiddly. Click events need to be routed to the correct layer, drag operations need thresholds to distinguish from clicks, and the default Mapbox interactions need to be replaced with custom controls.

The Right-Click Pan Problem

I wanted RTS-style controls: right-click to pan, middle-click to tilt and rotate. Mapbox's default is left-click drag to pan, which conflicts with unit selection. Disabling default controls and reimplementing them meant dealing with Mapbox's internal event system:

// Disable defaults
map.dragPan.disable();
map.dragRotate.disable();
map.doubleClickZoom.disable();

// Re-implement with right mouse button
map.on('mousedown', (e) => {
  if (e.originalEvent.button === 2) {
    startPan(e);
  } else if (e.originalEvent.button === 1) {
    startTiltRotate(e);
  }
});

Drag-Select vs Click

A 5-pixel movement threshold distinguishes clicks from drag-selects. Below the threshold, it's a click (select single unit or deselect). Above it, it's a rubber-band selection. A wasDrag flag prevents deck.gl from processing a click event after a drag-select completes.

What I'd Do Differently

If I were starting over, I'd invest more time upfront in a coordinate abstraction layer. Right now, different parts of the codebase work with different representations — {lat, lon} objects, [lon, lat] arrays (GeoJSON convention), {x, y} screen coordinates — and the conversions between them are a source of subtle bugs.

I'd also think harder about state ownership. The client currently polls the server and interpolates locally, which works but creates ambiguity about which side owns the "truth" during transitions. A more explicit state synchronization protocol would be cleaner.

But those are refinements. The core architecture — layered renderers, each doing what they're best at — has held up well. And watching a fleet of vehicles sweep across the map with fog of war lifting in their wake still makes me grin every time.
